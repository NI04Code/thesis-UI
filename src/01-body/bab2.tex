This chapter reviews the theoretical foundation and existing literature relevant to this research. It begins by exploring the underlying architecture of Kubernetes resource management and identifying the core operational challenge: the multi-dimensional resource fragmentation problem. To address this, the concepts of bin-packing and Multi-Criteria Decision Analysis (MCDA), specifically the Technique for Order Preference by Similarity to Ideal Solution (TOPSIS), are introduced. The discussion then focuses on the Kubernetes Descheduler framework, analyzing its current capabilities and inherent limitations. Finally, the chapter examines the need for complementary group eviction policies to support microservice architectures, ultimately setting the stage for the proposed policy-based descheduler framework.

\section{Kubernetes Architecture and Resource Management}
\label{sec:k8s_architecture}
%-----------------------------------------------------------------------------%
Kubernetes has emerged as the de-facto standard for container orchestration in cloud-native environments, providing robust abstractions for deploying, scaling, and managing containerized applications. 

At its core, Kubernetes operates using a master-worker architecture. Applications are deployed as Pods, which are the smallest deployable computing units consisting of one or more containers sharing network and storage namespaces. These Pods are scheduled to run on worker machines called Nodes. The Kubelet, an agent running on each Node, ensures that the containers described in the Pod specifications are running and healthy.

The initial placement of a Pod onto a Node is handled by the `kube-scheduler` component. The default scheduling process follows a two-step approach: filtering and scoring. During the filtering phase, the scheduler eliminates Nodes that cannot satisfy the Pod's constraints, such as inadequate CPU or memory availability, or unsatisfied hardware affinities. In the scoring phase, the scheduler ranks the remaining feasible Nodes using various scoring plugins to determine the optimal placement. 

For instance, the default `LeastAllocated` scoring strategy prioritizes Nodes with the fewest assigned resources to spread the workload evenly across the cluster. Conversely, the `MostAllocated` strategy favors Nodes that are already highly utilized, which effectively condenses workloads onto a smaller set of Nodes, fulfilling a basic bin-packing objective.

Furthermore, Kubernetes supports dynamic workload management through autoscaling mechanisms. The Horizontal Pod Autoscaler (HPA) and Vertical Pod Autoscaler (VPA) adjust the number of Pod replicas and resource requests based on utilization metrics. At the infrastructure level, the Cluster Autoscaler observes unschedulable Pods and automatically provisions new Nodes to accommodate them, or removes underutilized Nodes to reduce costs. However, because Pod assignments in Kubernetes are static (meaning a Pod is bound to a Node for its entire lifecycle until termination), the continuous scaling of workloads inevitably leads to degraded cluster states over time.

\section{Resource Fragmentation Problem}
\label{sec:resource_fragmentation}
%-----------------------------------------------------------------------------%
The static binding of Pods to Nodes in Kubernetes creates a persistent operational challenge known as Relative Resource Fragmentation (RRF)~\citep{arxiv_1506_07020}. This issue, often called the "Tetris problem" in cluster management, happens when pods of varying sizes are continuously created and destroyed. This constant activity leaves behind scattered, unusable gaps of resource capacity across different Nodes.

This phenomenon results in the formation of "capacity islands," a state where the total available resources across a cluster are adequate, but their physical dispersion makes them impossible to allocate to incoming workloads. Without active defragmentation, these unusable resource pockets significantly diminish cluster efficiency and drive up operational cloud costs~\citep{wiz_k8s_cost}.

Resource fragmentation in modern clusters is rarely one-dimensional; it requires multi-dimensional analysis, usually evaluating CPU and Memory concurrently. When workloads have asymmetric resource requirements, such as CPU-intensive web servers or memory-bound data caches, the cluster often suffers from stranded resources. Stranded capacity occurs when one resource dimension (e.g., CPU) on a Node is completely exhausted while the other (e.g., Memory) remains largely idle. A multi-dimensional fragmentation framework is therefore essential to detect and reconcile these imbalances, ensuring that both CPU and Memory are consumed at proportional rates.

\section{The Bin-packing Problem and Multi-Criteria Decision Analysis}
\label{sec:binpacking_mcda}
%-----------------------------------------------------------------------------%
Addressing resource fragmentation is fundamentally a variation of the classic bin-packing problem. In computer science, the bin-packing problem involves allocating a set of independent items with different weights into a finite number of bins such that the number of used bins is minimized. In the context of Kubernetes, the "items" are the Pods, and the "bins" are the cluster Nodes. 

Traditional scheduling algorithms often attempt to solve the bin-packing problem using single-weight heuristics, meaning they evaluate the placement based on a primary metric, such as total CPU availability. While simple to compute, this single-dimensional approach is inadequate for modern workloads that exhibit complex dependency on multiple resource dimensions, often exacerbating stranded capacity issues.

To overcome the limitations of single-weight bin-packing, Multi-Criteria Decision Analysis (MCDA) techniques can be applied to evaluate scheduling and eviction decisions across multiple dimensions simultaneously. One prominent MCDA method is the Technique for Order Preference by Similarity to Ideal Solution (TOPSIS). TOPSIS evaluates candidate alternatives against multiple criteria by determining the shortest geometric distance to a "positive ideal solution" (the best possible outcome across all metrics) and the longest distance from a "negative ideal solution" (the worst possible outcome)~\citep{arxiv_2506_04902}.

By applying TOPSIS to Kubernetes resource scheduling, operators can rank Pods or Nodes by simultaneously weighing their efficiency in utilizing CPU, Memory, and even specific hardware accelerators like GPUs. When used strategically in a consolidation pipeline, TOPSIS allows the system to identify not just any pod for eviction, but the specific pod that optimally restores proportional multi-dimensional balance to a fragmented Node.

\section{Kubernetes Descheduler and its Limitations}
\label{sec:k8s_descheduler}
%-----------------------------------------------------------------------------%
Because the `kube-scheduler` is inherently reactive (meaning it only makes decisions at the moment a Pod is created), it cannot correct the long-term entropy and resource fragmentation that develops within a running cluster. To provide corrective, proactive management, the Kubernetes ecosystem utilizes the Descheduler framework.

The Kubernetes Descheduler operates by periodically scanning the cluster state, identifying Pods that violate specific policies, and gracefully evicting them. Once evicted, these Pods are recreated by their respective controllers (e.g., ReplicaSets or Deployments) and placed into the scheduling queue, allowing the `kube-scheduler` to make newly informed placement decisions based on the current cluster topology. The Descheduler employs standard strategies to manage cluster health. For example, the `LowNodeUtilization` strategy aims to identify extremely underutilized Nodes and evict Pods from heavier Nodes, spreading the workload. Conversely, the `HighNodeUtilization` strategy attempts to free up underutilized Nodes by moving their few remaining Pods to more occupied Nodes, acting as an enabler for the Cluster Autoscaler to safely terminate empty instances.

Despite its utility, the standard Descheduler framework presents significant limitations when combating complex resource fragmentation. The built-in strategies predominantly rely on simple, single-dimensional thresholds (e.g., evicting if CPU usage is below 20\%). These strategies lack the sophistication required to perform multi-dimensional TOPSIS-based bin-packing evaluations. Consequently, while the standard Descheduler can enforce basic topology rules, it struggles to execute the highly targeted, multi-criteria consolidation required to eliminate advanced forms of multi-dimensional stranded capacity.

\section{Group Eviction for Microservices}
\label{sec:group_eviction}
%-----------------------------------------------------------------------------%
Beyond resource utilization metrics, proactive descheduling must also account for the architectural realities of distributed applications. The shift towards microservices has decoupled monolithic software into networks of highly interdependent components. In high-performance computing or low-latency environments, the physical proximity of these dependent services is critical for minimizing network overhead and maintaining latency contracts.

Current descheduling algorithms evaluate and evict workloads on a strictly per-pod basis. This individualistic approach poses a severe risk to service level agreements (SLAs). When one Pod belonging to an interdependent group is evicted for cluster balancing, its dependent partners remain on the original Node. This creates a "Split-Brain" state, temporarily forcing communication across network boundaries or even Availability Zones until the evicted Pod is successfully restarted and reconciled. 

To preserve the integrity of microservice latency contracts during proactive cluster consolidation, eviction logic must support application dependencies. This requires an "All-or-Nothing" (Atomic) group eviction mechanism. In an atomic eviction model, the system validates the evictability of an entire logical group of Pods, respecting multi-pod disruption constraints like Pod Disruption Budgets (PDB). This ensures that if one underlying component must be migrated for resource optimization, its dependent partners are migrated alongside it in a transactional manner.

\section{Current Approaches and the Research Gap}
\label{sec:research_gap}
%-----------------------------------------------------------------------------%
The literature reveals a prominent gap in current Kubernetes resource optimization capabilities. While the platform offers the Descheduler framework as a mechanism for post-placement workload migration, its default policies rely on rudimentary heuristics that process only 1 or 2 resource dimensions. This approach is insufficient for resolving the complex multi-dimensional resource fragmentation—the Tetris problem—that prevents optimal hardware consolidation. 

Furthermore, existing scheduling enhancements that introduce intelligent multi-dimensional evaluation (such as TOPSIS for bin-packing) are often disjointed from the descheduler lifecycle, missing the opportunity to use MCDA for targeted, proactive defragmentation. Finally, current automated eviction processes operate strictly at the individual pod level, posing severe "Split-Brain" latency risks for dependent microservices that lack atomic group migration capabilities. Addressing these limitations simultaneously is essential to form a cohesive, policy-based descheduler capable of advancing cluster FinOps efficiency without compromising application stability.
