This chapter reviews the concepts and prior work that form the foundation of this study: Kubernetes scheduling, resource fragmentation, multi-resource bin packing, MCDA/TOPSIS, the Kubernetes Descheduler, communication-aware scheduling, and the resulting research gap.

%----------------------------------------------------------------------------- %%
\section{Kubernetes Architecture and Resource Management}
\label{sec:kubernetesArchitecture}
%----------------------------------------------------------------------------- %%

Kubernetes manages containerized applications by distributing Pods across a set of worker Nodes. Each Pod declares resource requests and limits, commonly including CPU and memory. The kube-scheduler uses these declarations to determine which Nodes are feasible and then scores feasible Nodes to select a placement. This scheduling process is effective for initial placement, but it primarily reacts to each Pod at scheduling time and does not continuously optimize already-running Pods.

The scheduling framework generally consists of filtering and scoring phases. Filtering removes Nodes that cannot satisfy hard constraints such as resource availability, node affinity, taints, tolerations, and other placement requirements. Scoring then ranks the remaining feasible Nodes using configured priorities or plugins. Kubernetes also provides bin-packing-oriented scoring strategies such as \texttt{MostAllocated} and \texttt{RequestedToCapacityRatio}, which encourage placing Pods on more utilized Nodes to leave other Nodes emptier~\citep{k8s_resource_bin_packing}. However, these mechanisms are applied when Pods are scheduled; they do not automatically move existing Pods when the cluster becomes fragmented over time.

\section{Resource Fragmentation Problem}
\label{sec:resourceFragmentation}
%----------------------------------------------------------------------------- %%

Resource fragmentation occurs when available capacity is distributed across Nodes in a way that makes it difficult to place new workloads, even though aggregate capacity remains sufficient. This problem is often compared to the Tetris problem because each workload has a different multi-dimensional resource shape. A cluster may have enough total CPU and memory, but those resources may be split across Nodes in combinations that cannot fit new Pods~\citep{mishra_bellur_2016}.

Fragmentation is intensified by multi-resource imbalance. For example, a Node may have high CPU utilization but low memory utilization, while another Node has the opposite pattern. In this state, the cluster contains capacity that is technically free but practically stranded because each resource dimension cannot be used independently~\citep{xiao2013dynamic,guo2025joint}. Prior work has shown that directed Pod eviction can reduce this kind of fragmentation and improve resource utilization in Kubernetes clusters~\citep{larsson_klein_elmroth_2023}.

The operational impact of fragmentation is closely related to autoscaling. The Cluster Autoscaler can remove a Node only when its workloads can be safely moved elsewhere, and it may need to keep or add Nodes when Pods cannot fit into the currently available resource layout~\citep{cluster_autoscaler_faq}. Therefore, improving packing through controlled descheduling can support node reclamation and reduce unnecessary infrastructure usage.

\section{The Bin-Packing Problem and Multi-Criteria Decision Analysis}
\label{sec:binPackingMcda}
%----------------------------------------------------------------------------- %%

Cluster consolidation can be viewed as a multi-resource bin-packing problem. Nodes act as bins, while Pods act as items with CPU, memory, and possibly other resource dimensions. The objective is not simply to fill every Node, but to arrange workloads so that fewer Nodes are needed and remaining resources are less fragmented. Multi-resource packing has been studied in cluster schedulers such as Tetris, which considers multiple resource dimensions when placing workloads~\citep{grandl_tetris_2014}.

In Kubernetes descheduling, the challenge is slightly different from initial scheduling. The system must decide which already-running Pods should be evicted so that the subsequent scheduling process can improve the overall cluster state. A naive eviction strategy may remove Pods randomly or based on simple thresholds, but this may not target the Pods that contribute most to imbalance or stranded capacity.

Multi-Criteria Decision Analysis (MCDA) provides a structured way to rank alternatives using multiple criteria. TOPSIS is one MCDA method that ranks alternatives based on their distance from an ideal solution and a negative ideal solution~\citep{topsis_1981}. In this research, TOPSIS is used as a candidate approach for ranking Pods according to criteria such as resource usage, imbalance contribution, consolidation benefit, and network-aware considerations. This allows eviction to be treated as a strategic selection problem rather than a threshold-only decision.

\section{Kubernetes Descheduler and Its Limitations}
\label{sec:deschedulerLimitations}
%----------------------------------------------------------------------------- %%

The Kubernetes Descheduler is a component that evicts Pods based on configured policies so that the kube-scheduler can place them again under newer cluster conditions~\citep{k8s_descheduler_docs}. It is useful because Kubernetes does not generally move running Pods after they have been scheduled. Descheduling therefore provides a post-placement correction mechanism for problems such as imbalance, policy drift, or underutilization.

Existing Descheduler strategies are useful, but they are not a complete solution for the problem addressed in this study. First, many policies are based on thresholds or generic balancing rules and do not explicitly model which Pods are the best candidates for reducing multi-resource fragmentation. Second, after a Pod is evicted, final placement is still handled by the kube-scheduler, so the descheduling policy must carefully choose evictions that are likely to improve cluster state. Third, standard descheduling policies do not directly consider communication locality between dependent services.

These limitations motivate a custom Descheduler plugin or policy that combines fragmentation-aware source-node selection, strategic Pod ranking, and network-aware scoring or filtering. The goal is to use Descheduler as a consolidation trigger while preserving the kube-scheduler as the component responsible for final placement.

\section{Communication-Aware and Network-Aware Scheduling}
\label{sec:networkAwareScheduling}
%----------------------------------------------------------------------------- %%

Microservice applications often consist of multiple services that communicate frequently. If highly dependent services are placed far apart, especially across Nodes, zones, or high-latency links, application performance can degrade. Prior studies on network-aware Kubernetes placement argue that scheduling decisions should consider communication requirements, traffic volume, or network conditions rather than relying only on CPU and memory availability~\citep{marchese2022network,marchese_tomarchio_2022}.

Traffic-aware scheduling research similarly shows that service placement can benefit from considering changing communication patterns~\citep{chen2024trade}. Edge-oriented Kubernetes work such as Pogonip also demonstrates the importance of latency-aware placement for distributed applications in environments where network distance is significant~\citep{pusztai_pogonip_2021}.

For this research, network awareness is treated as a guardrail for consolidation. The proposed policy does not attempt to solve the entire traffic engineering problem. Instead, it uses network-aware scoring or filtering during eviction candidate selection so that Pods are not evicted when the expected rescheduling outcome is likely to worsen communication locality. This approach keeps the research focused on Descheduler-based consolidation while still addressing the latency risk introduced by network-blind eviction.

\section{Group Eviction, Atomic Migration, and Non-Atomic Scoring}
\label{sec:groupEviction}
%----------------------------------------------------------------------------- %%

One possible approach for maintaining communication locality is to move related Pods as a group. Group or gang-style migration can preserve dependency relationships by ensuring that related services are relocated together. However, atomic group movement can also make the packing problem harder because the cluster must find enough capacity for a larger combined workload. If the required capacity is unavailable, the whole movement may fail and the cluster remains fragmented.

For a consolidation-oriented Descheduler policy, non-atomic scoring can be more flexible. Instead of requiring a whole group to move together, each candidate Pod can be evaluated based on both its resource-fragmentation impact and its network-locality risk. This allows the policy to prefer candidates that improve packing while avoiding movements that are likely to cause significant latency degradation. Constraint-based pod packing work also indicates that packing decisions can be sensitive to constraint interactions and placement priorities~\citep{christensen2025priority}.

This study therefore adopts a score-based network-aware direction rather than a strictly atomic group-eviction direction. The policy can still consider service dependencies, but it treats them as scoring or filtering criteria instead of hard requirements that force all related Pods to be evicted together.

\section{Current Approaches and the Research Gap}
\label{sec:researchGap}
%----------------------------------------------------------------------------- %%

Prior work and existing Kubernetes mechanisms address parts of the problem, but not the complete combination targeted by this study. Kubernetes scheduling provides filtering and scoring mechanisms for initial placement, including bin-packing-oriented scoring strategies~\citep{k8s_resource_bin_packing}. The Descheduler provides a mechanism for post-placement correction~\citep{k8s_descheduler_docs}. Research on resource fragmentation and directed eviction shows that consolidation can reduce the number of Nodes needed for the same workload~\citep{larsson_klein_elmroth_2023}. Meanwhile, communication-aware scheduling research shows that network conditions and service dependencies can affect application performance~\citep{marchese_tomarchio_2022,chen2024trade}.

The gap is that existing approaches do not sufficiently combine these concerns inside a Descheduler-based eviction policy. A fragmentation-only policy may improve packing but worsen communication locality. A network-aware placement policy may improve latency but may not target resource fragmentation or autoscaler node reclamation. A threshold-based descheduler may rebalance utilization but may not strategically identify the Pods that most contribute to stranded capacity.

This research addresses that gap by proposing a Descheduler-based policy that ranks eviction candidates using MCDA/TOPSIS, targets resource fragmentation and imbalance, and incorporates network-aware scoring or filtering. The expected result is a consolidation strategy that improves packing and node reclaimability while reducing the risk of unnecessary latency degradation.
