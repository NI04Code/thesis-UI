\section{Research Background}
\label{sec:researchBackground}
%-----------------------------------------------------------------------------%
The shift toward microservices has decoupled application components to improve scalability and fault tolerance. However, this decoupling introduces network overhead. 
In high-performance computing (HPC) or low-latency systems, the physical placement of services that often communicate (e.g., a cache and its primary consumer) is critical to improve latency.
While the Kubernetes Scheduler supports "Inter-Pod Affinity" at creation, the Kubernetes Descheduler currently operates on a per-pod basis. 
This creates a "Fragmentation Gap": over time, node pressure or autoscaling events might evict one half of a low-latency pair, leaving the other behind. 
Current descheduling strategies lack "atomicity," meaning they cannot guarantee that a group of dependent services will be moved together to maintain their latency contract.


%-----------------------------------------------------------------------------%
\section{Problem Statement}
\label{sec:problemStatement}
%-----------------------------------------------------------------------------%
Existing Kubernetes descheduling mechanisms are individualistic. When a pod is evicted for balancing, its dependent partner remains, 
leading to temporary but significant latency spikes or "Split-Brain" performance degradation until the cluster reaches a new equilibrium.

%-----------------------------------------------------------------------------%
\subsection{Research Question}
\label{sec:researchQuestion}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Atomic Eviction Implementation:} How can we implement an All-or-Nothing (Atomic) eviction mechanism within the Kubernetes Descheduler framework to ensure transactional integrity?
    
    \item \textbf{Latency Impact:} To what extent does ``Group Descheduling'' reduce the duration of high-latency states compared to standard individual pod eviction?
    
    \item \textbf{Failure Handling:} How does the system handle eviction failures (e.g., PDB violations) to ensure no partial group moves occur, thereby maintaining application stability?
\end{itemize}

%-----------------------------------------------------------------------------%
\subsection{Research Limitation}
\label{sec:researchLimitation}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Infrastructure:} Limited to a small-scale cluster (3–5 nodes) on AWS EC2 us-east-2.

    \item \textbf{Network:} While testing across Availability Zones (AZs), we assume standard AWS networking without specialized hardware like SR-IOV.

    \item \textbf{Provisioning:} Using kubeadm to simulate production, but without enterprise-grade storage or complex service meshes (e.g., Istio) unless specifically required for latency measurement.
\end{itemize}

%-----------------------------------------------------------------------------%
\section{Research Objectives}
\label{sec:researchObjectives}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item Design a "Group Eviction" yaml metadata schema (labels/annotations) to define pod dependencies.

    \item Develop a custom Descheduler strategy that validates the "evictability" of an entire pods in a group and do “ All or Nothing” eviction.

    \item Evaluate the performance impact using a benchmark tool to compare group vs. individual eviction.
\end{itemize}


%-----------------------------------------------------------------------------%
\section{Research Contributions}
\label{sec:researchContributions}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Algorithm Design:} A transactional "All-or-Nothing" logic for pod evictions.

    \item \textbf{Open Source Descheduler Enhancement} Provide a new policy to evict group of pods instead of individual pods, enhancing kubernetes-sigs/descheduler configuration
\end{itemize}
