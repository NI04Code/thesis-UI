\section{Research Background}
\label{sec:researchBackground}
%-----------------------------------------------------------------------------%
Kubernetes has become the dominant orchestration platform for containerized workloads, but dynamic clusters still face a persistent packing problem: resource fragmentation. As Pods are created, scaled, and terminated over time, available CPU and memory become scattered into small residual gaps across different Nodes. Even when the cluster still has enough total capacity in aggregate, that capacity may no longer be usable for incoming workloads with larger or more imbalanced requests~\citep{mishra_bellur_2016,grandl_tetris_2014}.

This issue is not only conceptual; it has been observed directly in Kubernetes environments. Larsson \textit{et al.} show that as workloads evolve, Kubernetes clusters can enter fragmented states in which poor Pod-to-Node mappings strand resources, and they report that directed Pod eviction can reduce fragmentation enough to run the same workload using 16.7\% fewer Nodes than standard scheduling alone~\citep{larsson_klein_elmroth_2023}. Official Kubernetes documentation also acknowledges packing as a first-class scheduling concern through bin-packing-oriented scoring strategies such as \texttt{MostAllocated} and \texttt{RequestedToCapacityRatio}, but these mechanisms act only during placement and do not themselves repair fragmentation that has already accumulated over time~\citep{k8s_resource_bin_packing}.

Because Kubernetes binds a Pod to one Node for its lifetime unless the Pod is recreated, long-running clusters need a post-placement correction mechanism. The Descheduler provides that operational hook by evicting selected Pods so they can be placed again under a new cluster state~\citep{k8s_descheduler_docs}. This is especially relevant for consolidation, because the Cluster Autoscaler can remove Nodes only after workloads have been packed tightly enough to empty them safely~\citep{cluster_autoscaler_faq}. In other words, better packing is a prerequisite for reclaiming underutilized infrastructure rather than just adding more Nodes.

At the same time, modern cloud-native applications are increasingly composed of communicating microservices. For these workloads, a consolidation decision should not be judged solely by resource fit. Marchese and Tomarchio show that default Kubernetes scheduling does not account for application communication requirements or network conditions, motivating scheduling extensions that keep highly interacting services closer together~\citep{marchese_tomarchio_2022}. Similar network-aware placement ideas also appear in edge-oriented Kubernetes work such as Pogonip, which explicitly considers latency between communicating components~\citep{pusztai_pogonip_2021}. Therefore, fragmentation-aware consolidation should be complemented by dependency- or network-cost-aware eviction considerations so that improved packing does not unnecessarily worsen communication locality.

\section{Problem Statement}
\label{sec:problemStatement}
%-----------------------------------------------------------------------------%
The core issue in modern Kubernetes resource management is the absence of a proactive consolidation pipeline that can reduce fragmentation after the initial scheduling phase, while still respecting workload communication patterns. Specifically:
\begin{enumerate}
    \item \textbf{Static Resource Scatter:} The kube-scheduler does not explicitly detect when available CPU and memory have become too scattered across nodes to be practically useful.
    \item \textbf{Ineffective Balancing Heuristics:} Existing balancing approaches may treat resources too generically, making it difficult to identify pods that contribute most to stranded capacity or node-level imbalance.
    \item \textbf{Reactive Scale-Out:} Because fragmented resources cannot accommodate new Pods efficiently, the cluster may keep additional Nodes in service or trigger further scale-out even when aggregate capacity is still sufficient~\citep{larsson_klein_elmroth_2023,cluster_autoscaler_faq}.
    \item \textbf{Communication-Locality Risk:} Eviction decisions that focus only on packing can separate dependent services and increase communication overhead, especially in multi-node or multi-zone deployments.
\end{enumerate}

%-----------------------------------------------------------------------------%
\subsection{Research Question}
\label{sec:researchQuestion}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item How can a multi-stage descheduling pipeline be designed to transition a Kubernetes cluster from a fragmented state to a more consolidated and better packed state?
    \item What strategy can effectively identify source nodes for eviction and target nodes for consolidation so that underutilized nodes can be emptied and reclaimed by the Cluster Autoscaler?
    \item How can Multi-Criteria Decision Analysis (MCDA), such as TOPSIS-based ranking, be used to prioritize eviction candidates that contribute most to node-level inefficiency or resource imbalance?
    \item How can network-cost-aware or dependency-aware eviction considerations be incorporated so that fragmentation reduction does not unnecessarily worsen communication locality between related services?
\end{itemize}

%-----------------------------------------------------------------------------%
\subsection{Research Limitation}
\label{sec:researchLimitation}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Infrastructure:} Evaluation is limited to a small-scale Kubernetes cluster (3--5 nodes) on AWS EC2 in the us-east-2 region.
    \item \textbf{Network:} Experiments assume standard AWS networking without specialized hardware acceleration such as SR-IOV.
    \item \textbf{Provisioning:} The cluster is provisioned using kubeadm to approximate production behavior, but excludes enterprise-grade storage features and complex service meshes unless specifically needed for the evaluation scenario.
    \item \textbf{Framework Dependencies:} The solution is implemented within the Kubernetes Descheduler framework; it determines which pods should be evicted, while final placement remains the responsibility of the kube-scheduler.
    \item \textbf{Destructive Migration:} The study uses standard Kubernetes eviction and restart behavior rather than live migration.
    \item \textbf{Scope of Network Awareness:} Dependency or communication locality is treated as a complementary consideration for eviction filtering, not as a complete traffic-engineering solution.
\end{itemize}

%-----------------------------------------------------------------------------%
\section{Research Objectives}
\label{sec:researchObjectives}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Develop a fragmentation-aware consolidation strategy:} Design a multi-stage descheduling pipeline that identifies cluster-wide resource scatter, selects underutilized source nodes, and supports workload consolidation onto denser target nodes.
    \item \textbf{Create an imbalance-detection mechanism:} Develop a plugin or policy component that detects stranded resources and identifies pods that contribute to node-level CPU-memory imbalance.
    \item \textbf{Implement a strategic eviction engine:} Use MCDA, particularly TOPSIS-based ranking, to prioritize eviction candidates that provide the greatest benefit for restoring balance or emptying nodes.
    \item \textbf{Incorporate communication-locality awareness:} Add dependency-aware or network-cost-aware considerations so consolidation actions can avoid unnecessary degradation in inter-service communication patterns.
    \item \textbf{Evaluate consolidation outcomes:} Measure the effect of the proposed approach on packing efficiency, node reclaimability, and selected communication-related indicators under the experiment setup.
\end{itemize}


%-----------------------------------------------------------------------------%
\section{Research Contributions}
\label{sec:researchContributions}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item \textbf{Proactive Consolidation Pipeline:} A descheduler-based pipeline for reducing fragmentation and improving workload packing after the initial scheduling phase.
    \item \textbf{Enhanced Node Utility for Autoscaling:} A method to systematically empty underutilized nodes so that Cluster Autoscaler node reclamation becomes more feasible.
    \item \textbf{Smarter Offender Identification:} A TOPSIS-based ranking approach to identify eviction candidates that contribute most to stranded capacity or node imbalance.
    \item \textbf{Communication-Aware Eviction Consideration:} An extension that incorporates dependency locality or network-cost-aware filtering to reduce the risk of worsening inter-service communication.
    \item \textbf{Operational FinOps Relevance:} A practical implementation that targets stranded capacity reduction and more efficient infrastructure utilization in Kubernetes clusters.
\end{itemize}

Figure~\ref{fig:research_position} positions this research between two adjacent streams of work. The first stream focuses on improving cluster packing and reducing stranded capacity through bin-packing or defragmentation-oriented scheduling and descheduling. The second stream focuses on communication-aware placement so that interacting microservices remain in favorable network positions. This thesis occupies the overlap between those streams by treating fragmentation-aware consolidation as the main problem while using communication locality as a guardrail for eviction decisions. The goal is not to solve full traffic engineering, but to avoid consolidation actions that improve resource packing at the cost of clearly worse service placement.
