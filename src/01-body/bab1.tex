\section{Research Background}
\label{sec:researchBackground}
%-----------------------------------------------------------------------------%
Kubernetes has become the standard for orchestrating containerized workloads, yet it faces a persistent challenge known as the "Tetris problem" or Relative Resource Fragmentation (RRF). In dynamic clusters, pods arrive and terminate stochastically, leaving behind awkward, non-contiguous gaps in node capacity that are too small to host new, larger requests~\citep{arxiv_1506_07020}.

While the default scheduler handles the initial placement, it is inherently reactive and often unable to fix the long-term entropy of the cluster. This "resource scatter" prevents the Cluster Autoscaler from reclaiming nodes, leading to "capacity islands" where resources are technically available but functionally unusable. Solving this requires an active, cluster-wide consolidation strategy that identifies underutilized nodes and systematically migrates workloads to enable hardware reclamation, thereby maximizing cluster density and reducing unnecessary infrastructure costs~\citep{wiz_k8s_cost}.

Additionally, the shift toward microservices has decoupled application components to improve scalability, introducing network overhead. In high-performance computing (HPC) or low-latency systems, the physical placement of services that often communicate is critical. Current descheduling strategies operate on a per-pod basis and lack "atomicity," meaning they cannot guarantee that a group of dependent services will be moved together to maintain their latency contract. This creates a "Fragmentation Gap," leaving behind orphaned pods.

\section{Problem Statement}
\label{sec:problemStatement}
%-----------------------------------------------------------------------------%
The core issue in modern Kubernetes resource management is the lack of a proactive consolidation pipeline within the default orchestration layer, compounded by individualistic descheduling mechanisms. Specifically:
\begin{enumerate}
    \item Static Resource Scatter: The kube-scheduler lacks the "forward-looking" capability to identify that resources are too scattered to be useful.
    \item Ineffective Balancing Heuristics: Standard balancing tools often treat all resources equally, failing to identify specific "offender" pods that cause resource stranding (where one resource is exhausted while another remains idle).
    \item Reactive Scale-Out (Autoscaling Inefficiency): Because fragmentation scatters available resources into unusable chunks, new pods often cannot fit on existing nodes. This forces the Cluster Autoscaler to provision redundant new Virtual Machines (VMs) as new nodes in the cluster (a "scale-out" event). These redundant scale-out events not only incur unnecessary infrastructure costs~\citep{wiz_k8s_cost} but also increase the overall carbon footprint and energy consumption of the environment, even when the aggregate cluster capacity is mathematically sufficient~\citep{arxiv_2506_04902}.
    \item Latency Spikes during Eviction: When a pod is evicted for balancing, its dependent partner remains, leading to temporary but significant latency spikes or "Split-Brain" performance degradation.
\end{enumerate}

%-----------------------------------------------------------------------------%
\subsection{Research Question}
\label{sec:researchQuestion}
%-----------------------------------------------------------------------------%
Based on the problem statement described previously, the research questions addressed in this study are as follows:
\begin{itemize}
    \item Atomic Eviction Implementation: How can we implement an All-or-Nothing (Atomic) eviction mechanism within the Kubernetes Descheduler framework to ensure transactional integrity?
    \item Latency Impact: To what extent does ``Group Descheduling'' reduce the duration of high-latency states compared to standard individual pod eviction?
    \item Failure Handling: How does the system handle eviction failures (e.g., PDB violations) to ensure no partial group moves occur, thereby maintaining application stability?
    \item How can a multi-stage descheduling pipeline be designed to transition a cluster from a scattered state to a consolidated, "bin-packed" state?
    \item What strategy can effectively identify source nodes for eviction and target nodes for consolidation to ensure the Cluster Autoscaler can successfully reclaim hardware?
    \item How can Multi-Criteria Decision Analysis (MCDA) be utilized to rank pod eviction candidates, ensuring that the pods most responsible for node-level inefficiency are prioritized for migration?
\end{itemize}

%-----------------------------------------------------------------------------%
\subsection{Research Limitation}
\label{sec:researchLimitation}
%-----------------------------------------------------------------------------%
To maintain focus and define the scope of the study, the research limitations are established as follows:
\begin{itemize}
    \item Infrastructure: Limited to a small-scale cluster (3--5 nodes) on AWS EC2 us-east-2.
    \item Network: While testing across Availability Zones (AZs), we assume standard AWS networking without specialized hardware like SR-IOV.
    \item Provisioning: Using kubeadm to simulate production, but without enterprise-grade storage or complex service meshes (e.g., Istio) unless specifically required for latency measurement.
    \item Framework Dependencies: The solution is built as a plugin for the Kubernetes Descheduler Framework; it identifies what to move, but relies on the kube-scheduler to decide the new optimal location.
    \item Destructive Migration: Unlike live-migration prototypes (e.g., KubeDSM), this implementation uses standard Kubernetes evictions, which involve pod termination and restart.
    \item Scheduling Latency: Rapid, aggressive defragmentation may introduce temporary scheduling overhead or minor application downtime during the restart phase.
\end{itemize}

%-----------------------------------------------------------------------------%
\section{Research Objectives}
\label{sec:researchObjectives}
%-----------------------------------------------------------------------------%
In alignment with the formulated research questions, the objectives of this study are as follows:
\begin{itemize}
    \item Design a "Group Eviction" yaml metadata schema (labels/annotations) to define pod dependencies.
    \item Develop a custom Descheduler strategy that validates the "evictability" of an entire pods in a group and do ``All or Nothing'' eviction.
    \item Evaluate the performance impact using a benchmark tool to compare group vs. individual eviction.
    \item Develop the BinPackingFragmentation Consolidation Strategy: Implement a pipeline that computes cluster-wide scatter, filters underutilized "source" nodes, and cordons them to force the scheduler to pack evicted pods onto denser "target" nodes.
    \item Create an "Imbalance Correction" Plugin: Develop the ResourceFragmentation plugin to detect nodes with stranded resources and identify pods that disrupt the proportional consumption of CPU and Memory.
    \item Implement a Strategic Eviction Engine: Use the TOPSIS algorithm~\citep{arxiv_2506_04902} not as a mere scoring tool, but as a strategic engine to rank pods by their "benefit" to the node---evicting those that, once moved, provide the fastest path to emptying a node or restoring balance.
\end{itemize}


%-----------------------------------------------------------------------------%
\section{Research Contributions}
\label{sec:researchContributions}
%-----------------------------------------------------------------------------%
\begin{itemize}
    \item Algorithm Design: A transactional "All-or-Nothing" logic for pod evictions.
    \item Open Source Descheduler Enhancement: Provide a new policy to evict group of pods instead of individual pods, enhancing kubernetes-sigs/descheduler configuration.
    \item Proactive Consolidation Pipeline: Introduction of a Cordon-and-Evict logic within the descheduler that prevents "ping-pong" scheduling and ensures workloads are forced into a bin-packed configuration.
    \item Enhanced Node Utility for Autoscaling: A method to systematically empty underutilized nodes, providing the specific conditions required for the Cluster Autoscaler to terminate instances and reduce cloud spend.
    \item Smarter Offender Identification: Moving beyond random eviction by using TOPSIS-based ranking to identify pods that are the "best" candidates for removal based on their resource footprint relative to node capacity.
    \item Resource Optimization Framework: A practical software implementation that directly targets the reduction of "stranded capacity." By maximizing resource utilization, this approach translates to immediate infrastructure cost reductions~\citep{wiz_k8s_cost} and operational energy savings~\citep{arxiv_2506_04902} in large-scale Kubernetes environments.
\end{itemize}
