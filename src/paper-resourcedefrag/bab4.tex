\section{Experiment and Evaluation}
\label{bab:4}
%-----------------------------------------------------------------------------%

This section evaluates \textit{ResourceDefragmentation} against the upstream \textit{HighNodeUtilization} strategy across five node-reclamation and fragmentation scenarios, and then ablates the policy's three internal mechanisms to attribute the observed behavior to individual components. All evaluation uses request-space Node state, so placement feasibility and CPU-memory shape, rather than runtime telemetry, drive every comparison.

\subsection{Experimental Setup}
\label{sec:defrag_experimental_setup}

The experiment uses one control-plane Node and six worker Nodes. Each worker provides \(2000\,m\) allocatable CPU and approximately \(811\,Mi\) allocatable memory. Pause containers reserve declared requests while contributing negligible runtime load, so in requests mode the declared layout equals the accounted layout and no metrics server is required. Because a packing scheduler would otherwise pack the workloads at deploy time and leave no fragmented starting state, each scenario is seeded deterministically by cordoning all workers and uncordoning them one at a time, then uncordoning all workers before the Descheduler runs.

The scheduler uses \textit{MostAllocated} and \textit{BalancedAllocation}; this packing profile is a precondition, since with the default \textit{LeastAllocated} profile the policy cannot consolidate. Both systems operate on identical initial workloads and namespace scope, system workloads are excluded from eviction, the drain threshold is \(0.40\), the target-utilization ceiling is \(0.90\), and the Descheduler is re-run until a pass evicts zero Pods or a five-pass cap is reached. Each policy-scenario cell is a single representative run; metrics are computed by an independent tool that reads only \texttt{kubectl get nodes/pods -o json}, so the numbers are comparable across policies and never derived from Descheduler logs.

The two systems under comparison are:

\begin{itemize}
    \item \textbf{HighNodeUtilization (HNU):} the upstream utilization-based baseline with CPU and memory thresholds of \(40\%\).
    \item \textbf{ResourceDefragmentation (RD):} the proposed resource-only policy using request-based Node state, RII/FSI-based source ordering, \textit{predictSchedulerTarget}, and projected target bin score as its eviction-candidate criterion.
\end{itemize}

\subsection{Evaluation Metrics}
\label{sec:defrag_metrics}

Each run is assessed through four complementary metrics that capture node reclamation, dimensional balance, operational cost, and convergence. Every metric operates on request-space Node state, so the cluster layout is described entirely by per-Node resource requests. The fundamental quantity shared by all metrics is the per-dimension utilization fraction of a worker Node \(n\):

\[
u_{n,R}
=
\frac{R_{\text{req}}(n)}
{R_{\text{alloc}}(n)},
\qquad
R \in \{\text{cpu}, \text{mem}\}
\]

where \(R_{\text{req}}(n)\) is the sum of resource \(R\) requested by all evictable experiment Pods on Node \(n\), and \(R_{\text{alloc}}(n)\) is its allocatable capacity for resource \(R\).

%-----------------------------------------------------------------------------%
\paragraph{Active Node Count}
%-----------------------------------------------------------------------------%
The Active Node Count (\(N_{\text{active}}\)) is the primary indicator of node reclamation. A worker Node is active only when it hosts at least one evictable experiment workload:

\[
N_{\text{active}} = \sum_{n=1}^{N} a_n,
\qquad
a_n =
\begin{cases}
1, & \text{if } \sum R_{\text{req}}(n) > 0 \\
0, & \text{otherwise.}
\end{cases}
\]

A reduction between the before- and after-snapshots means one or more Nodes were fully drained and became reclaimable for scale-down.

%-----------------------------------------------------------------------------%
\paragraph{Resource Stranding Score}
%-----------------------------------------------------------------------------%
Because defragmentation optimizes the geometric \emph{shape} of utilization rather than its absolute spread, the cluster stranding score \(S\) aggregates the absolute CPU-to-memory imbalance across all active Nodes:

\[
    S = \sum_{n \in N_{\text{active}}} \left|u_{n,\text{cpu}} - u_{n,\text{mem}}\right|.
\]

A Node with a large gap between its CPU and memory fractions strands the under-used dimension. A lower \(S\) indicates that CPU and memory fractions are more tightly aligned across the cluster, unlocking trapped capacity. Unlike a global standard deviation, \(S\) is insensitive to intentional differences in absolute load and reacts only to per-Node dimensional skew.

%-----------------------------------------------------------------------------%
\paragraph{Balanced Headroom}
%-----------------------------------------------------------------------------%
The Balanced Headroom (\(H_{balanced}\)) converts residual request space into a schedulability-oriented quantity by counting how many reference Pods, each requesting \(200\,m\) CPU and \(80\,Mi\) memory, still fit across the cluster after descheduling. Let \(A_{\text{cpu}}(n)\) and \(A_{\text{mem}}(n)\) denote the remaining requested CPU and memory space on Node \(n\):

\[
H_{balanced}
=
\sum_{n=1}^{N}
\min\!\left(
\left\lfloor
\frac{A_{\text{cpu}}(n)}{200\,m}
\right\rfloor,
\left\lfloor
\frac{A_{\text{mem}}(n)}{80\,Mi}
\right\rfloor
\right).
\]

Because the per-Node term is limited by the tighter dimension, a Node with abundant free CPU but little free memory contributes little headroom, which is precisely the stranding the policy aims to remove.

%-----------------------------------------------------------------------------%
\paragraph{Eviction Count and Convergence}
%-----------------------------------------------------------------------------%
The Eviction Count (\(E_{count}\)) measures the operational cost of a sweep as the total number of Pods relocated, read from the per-pass Descheduler logs. A policy is more efficient when it reaches the same structural improvement with fewer Pod movements, since every eviction disrupts a running workload. Each run is also checked for convergence, that the per-pass count reaches zero within the five-pass cap, and for safety, that no after-snapshot leaves a Pod Pending.

The stranding results are the core contribution: in the two scenarios where dimensional stranding is reducible, S2 and S4, ResourceDefragmentation cuts it by \(73\%\) (\(3.495 \rightarrow 0.961\)) and \(79\%\) (\(3.071 \rightarrow 0.633\)) by using the predicted-target score to merge complementary CPU-heavy and memory-heavy shapes, while HNU leaves the layout untouched, as Figure~\ref{fig:defrag_stranding} makes visible. This gap, large defragmentation gains exactly where the threshold baseline does nothing, is the central result.

The remaining scenarios delimit the policy honestly. S1 shows a small stranding \emph{cost}: packing six Nodes into two slightly skews the two survivors (\(0.045 \rightarrow 0.287\)), an acceptable and expected consequence of aggressive consolidation given that the absolute stranding stays negligible while four Nodes are reclaimed. S3 and S5 are already at an irreducible stranding floor, since their complementary shapes cannot be co-packed under the \(0.90\) ceiling; ResourceDefragmentation correctly holds \(S\) constant rather than churning, which is the intended behavior when no shape improvement is feasible. The policy therefore acts where stranding is reducible and stays inert where it is not.

\subsection{Ablation Study}
\label{sec:defrag_ablation}

To attribute the behavior above to individual mechanisms, the full policy is compared against three variants, each disabling exactly one component: \textbf{FirstFeasible} replaces the projected-bin-score selection with the first feasible target; \textbf{NoPackUpward} removes the pack-upward guard; and \textbf{NoOrder} drains candidates in arbitrary order instead of by the RII/FSI priority. As Figure~\ref{fig:defrag_ablation} shows, each mechanism earns its place, but on a different axis:

\begin{itemize}
    \item \textbf{RII/FSI ordering protects stranding quality.} Disabling it regresses S4 sharply, from \(0.633\) to \(1.581\), because handling drain candidates in the wrong order relocates Pods before the complementary target is available. Elsewhere the ordering does not change the converged \(S\), so its role is to preserve defragmentation quality in the shape-sensitive case.
    \item \textbf{Pack-upward controls churn.} NoPackUpward reaches the same stranding as Full but at a higher eviction cost (S1: \(12\) vs \(8\); S4: \(11\) vs \(6\)). Without the guard it also empties an extra Node in S3 (\(6 \rightarrow 4\) vs \(6 \rightarrow 5\)) and S5 (\(4 \rightarrow 3\) vs \(4 \rightarrow 4\)), but that node-count gain costs roughly double the evictions and yields no stranding improvement, so it is a disruption regression rather than a win.
    \item \textbf{The bin-score selector is low-churn selection.} FirstFeasible matches Full on stranding in every scenario, so the projected-bin-score criterion does not drive stranding on this cluster. Its measurable effect is efficiency: Full reaches the same S1 result with \(8\) evictions where FirstFeasible needs \(12\).
\end{itemize}

The correct reading is therefore that the stranding wins come from the consolidation-and-defragmentation pipeline as a whole, candidacy, ordering, ceiling, and pack-upward, beating the utilization-threshold baseline, while the predicted-target selector keeps the eviction count down. Because each component either protects stranding quality or reduces churn, the default keeps all three enabled; a documented \textit{disablePackUpward} knob can expose the higher-reclamation, higher-churn mode for operators who explicitly want maximum node reclamation.

Four of the five scenarios reach a zero-eviction pass within two or three passes without oscillation. S2 is the exception: it continues to make one single-Pod move per pass and reaches the five-pass cap without a zero pass, because the fragmented layout admits a long tail of marginally improving relocations after the initial bulk consolidation. In every scenario, including S2, the run remains safe: no after-snapshot contains a Pending Pod, all evicted Pods are controller-owned and eligible under the default evictor, infrastructure namespaces are excluded, and no eviction request is left outstanding. These observations validate the feasibility guard, target ceiling, and bounded execution loop, and indicate that the pass cap, rather than an oscillation, is what bounds the tail case.

\subsection{Resource Policy Findings}
\label{sec:defrag_findings}

The evaluation yields four main findings:

\begin{itemize}
    \item ResourceDefragmentation reduces dimensional stranding by \(73\%\) in S2 and \(79\%\) in S4 and reclaims Nodes in S1, S2, and S4, all layouts in which HNU performs no eviction at all.
    \item On layouts where stranding is already irreducible (S3, S5), the policy correctly declines unproductive evictions, trading at most one reclaimed Node for substantially lower churn.
    \item The ablation attributes each mechanism to a distinct role: RII/FSI ordering preserves stranding quality (S4), pack-upward bounds eviction churn, and the projected-bin-score selector lowers eviction count without changing stranding.
    \item Every run is safe, leaving no Pod Pending; four of five scenarios converge within three passes, with S2 bounded by the five-pass cap rather than oscillation.
\end{itemize}

\subsection{Threats to Validity}
\label{sec:defrag_threats}

The evaluation is designed to isolate the request-based defragmentation mechanism, and this focus introduces limitations that bound how far the results should be generalized.

\textbf{Construct validity.} The experiment describes each Node purely through the declared CPU and memory requests of its Pods, and workloads are pause containers that reserve those requests while producing negligible runtime load. This is a deliberate consequence of the request-based scope of \textit{ResourceDefragmentation}: because the kube-scheduler admits Pods on effective requests, request-space state is the correct signal for placement feasibility and bin-packing shape. It does mean, however, that the reported reclamation and stranding metrics reflect the declared-request layout rather than live consumption. Divergence between requests and actual usage, and the service-level cost of evicting a busy Pod, are outside the scope of this policy and are addressed by a separate actual-usage pre-eviction guard rather than measured here.

\textbf{Internal validity.} Each policy-scenario cell is a single representative run rather than a multi-seed average, so small per-run variation is not quantified; every run is nonetheless checked for convergence and for the absence of Pending Pods. The ablation in Section~\ref{sec:defrag_ablation} isolates the contribution of each component by disabling one mechanism at a time under identical conditions, which strengthens the attribution of the observed effects. The comparison against external systems still uses a single baseline, \textit{HighNodeUtilization}, because it is the upstream, in-tree consolidation strategy and therefore a faithful production reference; reimplementing prior proposals that publish only pseudocode, without released code or artifacts, would risk an unfaithful comparison whose behavior reflects our reconstruction rather than the original design.

\textbf{External validity.} All experiments run on a single small cluster of one control-plane Node and up to six homogeneous AWS EC2 worker Nodes, each with an identical CPU and memory profile. The five scenarios are hand-constructed to exercise specific fragmentation shapes rather than sampled from production traces, so they demonstrate the mechanism under controlled conditions but are not a representative distribution of real cluster layouts. Larger clusters, heterogeneous Node capacities, multi-tenant workloads, and additional resource dimensions such as storage and GPU may change both the magnitude of the gains and the eviction cost, and evaluating the policy at scale is left to future work.

\textbf{Scheduler alignment.} \textit{ResourceDefragmentation} decides which Pods to evict, but final binding remains the responsibility of the kube-scheduler, and the evaluation confirmed that the intended consolidation is realized only under the \textit{MostAllocated} and \textit{BalancedAllocation} packing profile. The predicted target is an estimate of where a replacement will land; if the active scheduler profile is not aligned with the policy's intent, an evicted Pod may be placed on a Node other than the predicted target, which can reduce the realized defragmentation quality.
